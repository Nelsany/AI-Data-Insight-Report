# PRD（标准版）— AI 驱动的评论数据导入→痛点分析→HTML 报告一体化产品（首期：数据导入）

> 文档目的：用于产品评审、研发排期、测试验收与后续迭代对齐  
> 版本：v1.1  
> 状态：Draft  
> 目标用户：品牌/市场/增长；电商运营/产品经理  
> 首期平台：淘宝  
> 首期类目：香水、口红、面膜  

---

## 0. 一句话定义
用户在一个前端网站里通过自然语言输入任务，并上传已整理好的评论数据文件（SQL 导出/Excel/CSV 等），系统自动完成 **数据导入 → 痛点归类与解释 → 生成含图表的 HTML5 报告**，并直接输出可访问的分享链接。

---

## 1. 背景、问题与机会

### 1.1 背景
品牌与电商团队需要更高频、更低成本地理解消费者反馈，包括好评卖点、差评痛点、使用场景、竞品对比、包装/物流/客服等非商品因素，从而服务于选品、内容、投放、详情页优化和产品改进。

### 1.2 当前问题
1. **数据获取成本高**：评论抓取通常依赖技术同学、外包或零散脚本，需求响应慢。
2. **分析过程不标准**：不同人员使用不同标签和口径，导致结果难复盘、难横向比较。
3. **报告交付割裂**：原始数据、分析结果、图表和结论分散在多个工具里，产出效率低。
4. **结果可信度不足**：很多结论只有总结，没有代表性原文和样本统计，难以说服业务团队。

### 1.3 产品机会
将“数据导入、分析、报告”封装成同一个前端网站中的任务型体验，让非技术用户也能通过一句自然语言完成消费者洞察闭环。

## 2. 产品目标与范围

### 2.1 首期产品目标
**G1：一句话创建任务**  
- 用户输入自然语言即可生成结构化任务，并进入导入数据流程。

**G2：支持类目选择 + AI 自动识别**  
- 用户可手动选择类目，也可由 AI 自动识别大类，并允许编辑确认。

**G3：完成评论数据导入闭环（MVP）**  
- 首期不做任何平台爬虫，用户上传已拉取的数据（CSV/XLSX/JSON；SQL 导出建议转 CSV）。
- 系统完成字段校验、去空行、基础清洗，并入库供分析使用。

**G4：完成痛点/卖点分析并可解释**  
- 输出负面痛点 TOP N、正面卖点 TOP N，并附代表性原文证据与占比。

**G5：生成 HTML5 报告并默认产出分享链接**  
- 不做团队空间，首期报告默认以可访问分享链接形式交付。

### 2.2 首期非目标
- 不做任何平台爬虫（淘宝/天猫/抖音/小红书等全部不做）。
- 不做登录态、Cookie、验证码处理等抓取相关能力。
- 不做团队协作空间、组织权限体系。
- 不做自动 SQL 连接器（首期只接文件导入；SQL 导出由用户在外部完成）。

## 5. 端到端用户流程

### 5.1 主流程
1. 用户输入自然语言任务（想分析什么、类目、关注点、样本范围等）。
2. AI 解析出结构化参数（类目/关键词/分析重点/样本量等）。
3. 用户上传评论数据文件（CSV/XLSX/JSON；SQL 导出建议转 CSV）。
4. 系统导入并校验数据（字段映射、去空行、基础清洗）。
5. 用户点击开始分析（或导入后自动开始）。
6. 系统进入分析流程，提炼痛点/卖点并生成报告。
7. 系统生成 HTML 报告并返回分享链接。

### 5.2 类目识别交互建议
- AI 返回结构化字段：`平台`、`类目`、`关键词`、`样本量`、`时间范围`、`分析重点`
- 用户可在提交前编辑这些字段。
- 当类目识别置信度低时，必须提示用户确认。

### 5.3 首期输入方式
首期仅支持：
- **文件导入**：CSV / XLSX / JSON（SQL 导出建议转 CSV）

首期不支持：
- 任何平台爬虫与抓取入口（关键词搜索/商品链接/店铺链接均不做）

## 6. 功能需求

### 6.1 Step 1：导入数据（替代爬虫）

#### 6.1.1 自然语言创建任务
用户输入自然语言后，系统需解析出以下字段：
- `platform=淘宝`
- `category`
- `keyword`
- `sample_size`
- `time_range`
- `rating_filter`
- `analysis_focus`

示例：
“分析淘宝香水评论，分析差评原因并出报告”

解析结果：
- 平台：淘宝
- 类目：香水
- 关键词：香水
- 样本量：默认值
- 时间范围：默认值
- 分析重点：差评原因

#### 6.1.2 类目选择与自动识别
首期支持类目：
- 香水
- 口红
- 面膜

交互要求：
- 用户可手动选择类目。
- AI 可自动识别类目。
- AI 识别结果必须可编辑。

#### 6.1.3 文件导入模式（MVP）
用户上传数据文件，系统解析并入库。

支持格式：
- CSV / TSV（建议 UTF-8）
- XLSX / XLS
- JSON（数组，每个元素为一条评论记录）

SQL 支持说明（首期）：
- 用户可在外部系统（如 BI、数仓、数据库客户端）执行 SQL 导出评论表，并保存为 CSV/XLSX 后上传。
- 首期不解析 .sql 脚本、不直连数据库。

#### 6.1.4 导入执行与展示
任务详情页需展示：
- 当前阶段
- 已导入评论条数
- 目标评论条数（用于限制分析样本）
- 导入日志
- 预估剩余时间（可选）

可操作项：
- 重试
- 取消

#### 6.1.5 数据预览
导入完成后需展示抽样评论预览，字段建议包含：
- 评论时间
- 评分
- 评论正文
- 追评内容
- SKU/规格（若可得）
- 点赞数（若可得）

字段映射（允许中英文列名）：
- 内容：content / 正文 / 内容 / 评论
- 评分：rating / 评分 / 星级
- 时间：date / commentAt / 时间 / 日期
- 追评：appendContent / 追评 / 追加评论
- 规格：sku / 规格 / 型号 / 色号
- 点赞数：likeCount / 点赞数 / 赞
- 商品链接（可选）：productUrl / 商品链接
- 商品标题（可选）：productTitle / 商品标题

#### 6.1.6 限制与约束
- 首期设置单任务**导入/分析样本上限**，建议默认控制在 `2000` 条以内（可配置）。
- 上传文件大小限制（建议 30MB）与格式校验（仅 CSV/XLSX/JSON）。
- 重新上传默认覆盖该任务下已导入的数据（MVP 行为）。

---

### 6.2 Step 2：分析数据

#### 6.2.1 分析输出
首期报告需至少输出：
1. 评价分布
2. 负面痛点 TOP10
3. 正面卖点 TOP10
4. 每个主题的代表性原文
5. 核心关键词/高频词

#### 6.2.2 分析方法建议
采用“规则/字典 + LLM 辅助 + 聚类兜底”的组合方式：
1. 文本清洗
2. 情感判断
3. 方面抽取
4. 映射到类目标签体系
5. 聚合统计
6. 代表性证据筛选

#### 6.2.3 类目标签体系
首期建议分别建立三套基础标签字典：

**香水**
- 留香
- 味道
- 刺鼻/不适
- 包装
- 性价比
- 物流

**口红**
- 显色
- 掉色/沾杯
- 滋润度/拔干
- 色差
- 包装
- 性价比

**面膜**
- 补水
- 刺激/过敏
- 精华液多少
- 服帖度
- 气味
- 性价比

#### 6.2.4 可解释性要求
每个痛点/卖点需展示：
- 标签名称
- 一句话总结
- 占比
- 样本量
- 代表性原文

### 6.3 Step 3：生成报告

#### 6.3.1 报告形态
- 报告为 HTML5 单页。
- 默认生成可访问分享链接。
- 首期不区分私有/团队空间权限。

#### 6.3.2 报告结构
1. 摘要
2. 数据概览
3. 评价分布图
4. 负面痛点 TOP10
5. 正面卖点 TOP10
6. 关键词/主题分布
7. 方法说明
8. 免责声明

#### 6.3.3 图表清单
- 评价分布图
- 痛点占比条形图
- 卖点占比条形图
- 关键词词云或高频词列表

#### 6.3.4 报告分享
首期规则：
- 生成后直接产出分享链接
- 不做团队空间
- 不做复杂权限模型

## 7. AI 能力设计

### 7.1 首期能力边界
首期不建议（也不允许）让 AI 生成任意爬虫代码并执行。首期实现方式为：
- AI 负责理解用户意图
- AI 负责识别类目与生成参数
- 数据获取由用户在外部系统完成（SQL/Excel/BI 导出），平台侧风险与合规由用户自行承担

### 7.2 能力拆分
1. **任务解析器**
   - 将自然语言转成结构化参数
2. **类目识别器**
   - 识别香水/口红/面膜
3. **导入解析器**
   - 解析 CSV/XLSX/JSON 并做字段映射、基础清洗与入库
4. **分析器**
   - 完成分类、聚类、摘要与证据选择
5. **报告生成器**
   - 将结构化分析结果渲染为 HTML 报告

### 7.3 输出约束
- LLM 输出必须经过 JSON Schema 校验。
- 类目字段必须限制在首期支持类目中。
- 样本量必须受控，不允许无限导入/无限分析。

## 9. 技术方案建议

### 9.1 架构建议
1. 前端负责任务创建、进度展示和报告访问。
2. 后端负责任务编排、数据导入/校验、分析调度和报告生成。
3. 导入与分析采用异步任务机制（首期可简化为“导入成功后入队分析”）。
4. 报告生成后可托管为静态 HTML 页面。

### 9.2 前端重点
- 三步流程可视化
- 任务状态实时更新
- 类目识别结果可编辑
- 报告页支持图表与证据展开

### 9.3 后端重点
- 文件导入解析与字段映射
- 异步任务与失败重试机制
- 分析结果可追溯
- 报告静态化与分享链接生成

## 10. 合规、风控与安全

### 10.1 合规边界
- 数据仅用于聚合分析与报告输出。
- 不做个人画像。
- 不输出个人敏感信息。

### 10.2 数据最小化
- 不采集昵称、头像、联系方式等个人信息。
- 评论原文仅作分析证据使用。

### 10.3 数据导入风控策略
首期不做爬虫/抓取，主要风险转为“文件上传与数据安全”：
- 文件类型白名单（CSV/XLSX/JSON）与大小限制
- 解析失败需要给出清晰错误信息（缺少内容列/编码问题/空文件等）
- 数据落库前进行基础清洗（去空行、字段类型校验）

### 10.4 安全治理
若后续引入“AI 生成脚本并执行”的能力，必须配套：
- 域名白名单
- 静态扫描
- 资源限制
- 运行审计

## 15. 方案风险与建议
1. **数据导入质量不一**：用户导出的列名/编码/时间格式可能不同。建议提供字段模板、错误提示与映射规则说明。  
2. **类目自动识别必须允许人工修正**：否则会直接影响分析标签命中率。  
3. **报告必须提供证据引用与样本统计**：否则业务方很难信任 AI 的结论。  
4. **首期范围收敛是正确策略**：先做淘宝 + 文件导入 + 3 个类目，有利于快速验证闭环价值。
